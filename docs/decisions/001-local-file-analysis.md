# ADR-001: 로컬 파일 기반 분석 방식 채택

## 상태

채택됨 (Accepted)

## 맥락

Bart Dashboard는 Claude Code CLI로 개발 중인 프로젝트의 진행 상황을 시각화합니다. 프로젝트 데이터(Plan.md 진행률, Git 커밋, 기술 스택, ERD)를 어떻게 수집하고 저장할지 결정해야 합니다.

고려한 대안:
1. **데이터베이스에 분석 결과 저장** (SQLite, PostgreSQL 등)
2. **로컬 파일 시스템을 매 요청마다 직접 분석**
3. **인메모리 캐시 + 파일 감시 기반 갱신**

## 결정

**로컬 파일 시스템을 매 요청마다 직접 분석**하는 방식을 채택합니다.

프로젝트 등록 정보(ID, 이름, 경로)만 `data/projects.json` 파일에 저장하며, 진행률, 커밋 이력, 기술 스택 등 모든 분석 데이터는 요청 시점에 실시간으로 파일 시스템과 Git 명령을 통해 수집합니다.

## 근거

### 설치 부담 최소화

- 데이터베이스를 설치하고 설정하는 과정 없이, `npm install`만으로 바로 사용할 수 있습니다.
- 개인 개발자 도구로서, 가능한 한 적은 외부 의존성이 중요합니다.

### 항상 최신 상태 보장

- 파일을 직접 읽으므로, 데이터 동기화 문제가 발생하지 않습니다.
- 별도의 "데이터 갱신" 작업 없이도 항상 최신 상태를 보여줍니다.
- Git 커밋, Plan.md 수정 등의 변경이 즉시 반영됩니다.

### 데이터 불일치 방지

- 데이터베이스를 사용하면 실제 파일 상태와 DB 데이터 간 불일치가 발생할 수 있습니다.
- 로컬 파일 직접 분석은 이러한 불일치 가능성을 원천적으로 제거합니다.

### 단순한 배포와 이동

- 프로젝트 폴더를 다른 컴퓨터로 복사하면 그대로 동작합니다.
- 데이터베이스 마이그레이션이나 데이터 이전이 필요 없습니다.

## 결과

### 장점

- 설치가 간단합니다 (`npm install`만으로 충분).
- 데이터가 항상 파일 시스템의 실제 상태와 일치합니다.
- 데이터베이스 관련 장애점이 없습니다.
- 프로젝트 이동/복사가 자유롭습니다.

### 단점

- **응답 속도**: 매 요청마다 파일을 읽고 Git 명령을 실행하므로, 대규모 프로젝트에서는 응답이 느릴 수 있습니다.
  - 완화 방안: `getProjectSummary`에서 Plan, Git 상태, 최근 커밋을 `Promise.all`로 병렬 실행
- **이력 추적 불가**: 시간에 따른 진행률 변화 추이를 저장하지 않습니다.
  - 향후 필요 시 경량 시계열 저장(SQLite 등)을 추가할 수 있습니다.
- **대규모 Git 저장소**: 커밋 수가 매우 많은 경우 `git log` 실행 시간이 길어질 수 있습니다.
  - 완화 방안: `DEFAULT_GIT_LOG_LIMIT = 50`으로 최근 커밋만 조회

## 관련 파일

- `backend/src/services/projectService.ts` - projects.json 기반 저장/조회
- `backend/src/analyzers/gitAnalyzer.ts` - Git 명령 직접 실행
- `backend/src/parsers/planParser.ts` - Plan.md 직접 읽기/파싱
- `backend/src/analyzers/stackDetector.ts` - 파일 시스템 직접 탐색
